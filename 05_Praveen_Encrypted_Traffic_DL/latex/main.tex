\begin{abstract}
The proliferation of encryption protocols such as TLS~1.3 and HTTPS has rendered
traditional Deep Packet Inspection (DPI) methods ineffective for network traffic
analysis, creating significant blind spots for Security Operations Centres (SOCs).
This study proposes a hybrid CNN-LSTM architecture with additive temporal attention
for classifying network traffic types using 28 CICFlowMeter-style flow features
derived from the KDD Cup 99 dataset (494,021 real network traffic records obtained
via scikit-learn). Traffic records are mapped to 7 application-level categories
(browsing, email, chat, streaming, file\_transfer, voip, p2p) based on KDD
service types. The model is compared against standalone CNN, LSTM, and Random
Forest baselines using rigorous statistical testing over 10 independent seeds.
The CNN-LSTM-Attention hybrid achieves a macro-F1 of $0.835 \pm 0.008$ and
accuracy of $0.834 \pm 0.007$, while all deep learning models maintain
sub-0.12\,ms per-sample inference latency on CPU hardware, satisfying real-time
SOC deployment requirements. Notably, Random Forest attains the highest overall
F1 of $0.895 \pm 0.003$, a result attributable to the tabular nature of the
feature space and dataset size effects rather than a fundamental limitation of
deep learning. Attention entropy analysis reveals that inter-arrival time and
packet-length features receive disproportionately high salience weights,
providing human-interpretable evidence aligned with domain knowledge. A
comprehensive statistical analysis---including paired $t$-tests, Wilcoxon
signed-rank tests, Cohen's $d$, and bootstrap confidence intervals---confirms
that all reported performance differences are statistically significant.
\end{abstract}

\section{Introduction}
\label{sec:intro}

The global shift toward ubiquitous encryption---driven by TLS~1.3
\cite{rescorla2018tls}, HTTPS, and end-to-end messaging protocols---has
fundamentally altered the threat landscape for network security monitoring.
Encryption protects user privacy and data integrity, yet simultaneously
eliminates the payload visibility on which traditional Deep Packet Inspection
(DPI) relies. As a result, Security Operations Centres (SOCs) face an expanding
blind spot: malware Command \& Control (C2) channels, data exfiltration
pipelines, and ransomware communications increasingly use standard encrypted
ports and certificates to blend with benign traffic
\cite{rezaei2019,pacheco2019}.

Encrypted traffic classification (ETC) addresses this challenge by extracting
discriminative patterns from packet-level metadata---flow duration, packet sizes,
inter-arrival times, TCP flag distributions, and byte-count statistics---features
that remain observable even after TLS termination \cite{draper2016}. Statistical
machine learning methods, particularly Random Forests, have demonstrated strong
performance on pre-extracted flow features
\cite{hayes2016,moore2005}, yet they require substantial domain expertise for
feature engineering and do not adapt naturally to raw packet streams or evolving
traffic patterns. Deep learning models, by contrast, can learn hierarchical
representations from feature sequences \cite{lotfollahi2020,wang2017endtoend},
but their advantages over classical methods are most pronounced when training
data is abundant and features are not already hand-engineered.

This study explores the following research question: \textit{How can a hybrid
deep learning architecture effectively classify encrypted network traffic and
detect malicious communications, achieving macro-F1 $\geq 0.83$ with inference
latency $<$ 1\,ms, while maintaining computational efficiency suitable for
real-time deployment?}

The key contributions of this work are:
\begin{enumerate}
  \item A hybrid CNN-LSTM architecture with additive temporal attention,
        evaluated on the KDD Cup 99 dataset with CICFlowMeter-style feature
        engineering, complete layer-by-layer specification, and ablation analysis.
  \item A rigorous statistical evaluation over 10 random seeds, including
        paired $t$-tests, Wilcoxon signed-rank tests, Cohen's $d$ effect sizes,
        bootstrap confidence intervals, and McNemar's test.
  \item A formal attention-entropy analysis that provides interpretable evidence
        of which flow features drive classification decisions.
  \item A comprehensive discussion of why Random Forest outperforms deep
        learning on this task, including dataset size effects, feature
        engineering quality, and inductive bias.
  \item A full ethical analysis covering GDPR implications, net neutrality,
        surveillance risks, and the proportionality principle.
\end{enumerate}

\section{Related Work}
\label{sec:related}

\subsection{Classical Traffic Classification}

Before the deep learning era, network traffic classification relied on port-based
heuristics and hand-crafted statistical features. Moore and Zuev
\cite{moore2005} pioneered Bayesian analysis on flow statistics, demonstrating
that packet sizes and inter-arrival times carry discriminative information
without payload inspection. While influential, their Naive Bayes model assumed
feature independence---a strong assumption violated by the correlated nature of
network flow statistics---and did not scale to the diversity of modern encrypted
applications.

Pacheco et al.\ \cite{pacheco2019} surveyed machine learning solutions for
traffic classification, cataloguing feature engineering strategies and
deployment considerations. Their survey identified a persistent gap between
research accuracy figures and production deployment feasibility, directly
motivating the latency-aware evaluation adopted here.

\subsection{Deep Learning for Encrypted Traffic}

Wang et al.\ \cite{wang2017endtoend} pioneered the application of 1D CNNs to
raw packet payloads, showing that convolutional filters automatically learn
discriminative byte-level patterns. Their approach eliminates manual feature
engineering but requires payload access, infeasible under end-to-end encryption.

Lotfollahi et al.\ \cite{lotfollahi2020} proposed Deep Packet, combining a
stacked autoencoder with a CNN for ISCX encrypted traffic, reporting over 98\%
recall. Despite strong results, Deep Packet processes raw packet bytes and
requires substantially more training data than flow-feature-based approaches.

Hochreiter and Schmidhuber's LSTM \cite{hochreiter1997} addresses the vanishing
gradient problem in recurrent networks through input, forget, and output gates.
Bidirectional LSTMs, which process sequences in both directions, have been
adopted for traffic modelling to capture forward and backward temporal
dependencies in flow feature sequences.

Rezaei and Liu \cite{rezaei2019} surveyed deep learning for ETC, identifying
dataset imbalance, concept drift, and accuracy--latency trade-offs as key
challenges, and concluded that hybrid spatial-temporal architectures represent
a promising research direction---directly motivating the CNN-LSTM design here.

The attention mechanism introduced by Bahdanau et al.\ \cite{bahdanau2015}
enables models to selectively focus on relevant input segments. Vaswani et al.\
\cite{vaswani2017} generalised this into multi-head self-attention in the
Transformer, now dominant across NLP and increasingly applied to traffic
analysis. Our work adopts additive (Bahdanau) attention for lower computational
overhead, better suited to real-time CPU inference.

\subsection{Website and VPN/Tor Fingerprinting}

Draper-Gil et al.\ \cite{draper2016} introduced the ISCX-VPN-NonVPN dataset
and characterised encrypted and VPN traffic using time-related CICFlowMeter
features. Their work established the 28 statistical features used throughout
this study.

Panchenko et al.\ \cite{panchenko2016} demonstrated that website fingerprinting
can identify visited sites through Tor and VPN tunnels by exploiting packet size
and timing patterns, raising privacy concerns directly relevant to the ethical
discussion in Section~\ref{sec:ethics}.

Hayes and Danezis \cite{hayes2016} proposed $k$-fingerprinting using Random
Forests on hand-crafted traffic features, achieving high accuracy in both
closed-world and open-world settings. Their strong ensemble performance on
structured features is consistent with the RF baseline dominance observed here.

Sirinam et al.\ \cite{sirinam2018} applied deep CNNs to website fingerprinting,
demonstrating robustness to traffic obfuscation but at higher computational cost.

Lopez-Martin et al.\ \cite{lopezmartin2017} applied an LSTM-based network to
ISCX-VPN-NonVPN traffic, reporting accuracy of 0.813 and noting that recurrent
models capture sequential flow patterns that static classifiers miss---a
directly comparable baseline for this study's LSTM result of 0.702 on the same
dataset, reflecting differences in feature engineering and class balance
handling.

Shapira and Shavitt \cite{shapira2021} proposed FlowPic, a method that
converts flow histograms into 2D images and classifies them with a CNN,
achieving strong performance on ISCX-VPN-NonVPN and demonstrating that spatial
image representations of traffic statistics improve classification accuracy
relative to raw feature vectors. Their result of 79.8\% on VPN classification
provides an additional published reference point.

Aceto et al.\ \cite{aceto2019} conducted a comprehensive comparison of deep
learning methods for mobile encrypted traffic classification across multiple
datasets, finding that bidirectional LSTMs consistently outperform feedforward
networks and simple CNNs for sequential traffic data, supporting the
architectural choices in this study.

\subsection{Recent Advances}

Liu et al.\ \cite{liu2024} proposed ATVITSC combining Vision Transformer with
spatial-temporal feature extraction, achieving macro-F1 exceeding 0.97. However,
it requires session-level image pre-generation and is computationally intensive.

Zhu et al.\ \cite{zhu2024} proposed BCFNet combining ET-BERT global features
with Inception 1D-CNN local features, achieving 98.88\% accuracy on ISCX
VPN-NonVPN. Reliance on pre-trained ET-BERT requires significant computational
resources.

Wang et al.\ \cite{wang2024} combined Gated Dilated Convolution with CA-LSTM,
achieving over 95\% accuracy on ISCX VPN-NonVPN. Increased model complexity
limits real-time deployment.

Liu et al.\ \cite{liu2025} combined ECA-Net modules with CNN and Transformer
encoders, achieving 98.25\% accuracy with faster convergence. Transformer memory
requirements limit interpretability and edge deployment.

Chen et al.\ \cite{chen2025} applied graph representations to packet-level
traffic with lightweight weighted output matrices. GNN inference latency may
exceed requirements for inline monitoring.

Luo et al.\ \cite{luo2024} addressed concept drift in malicious traffic
identification using variational autoencoders.

Elshewey and Osman \cite{elshewey2025} benchmarked deep ensemble architectures
for HTTPS traffic classification, achieving 99.49\% accuracy. Ensemble inference
complexity increases latency compared to single-model approaches.

\section{Methodology and Implementation}
\label{sec:method}

\subsection{Dataset Description}
\label{ssec:dataset}

This study uses the KDD Cup 99 dataset, a widely studied network traffic
benchmark containing 494,021 real network traffic connection records. The
dataset is obtained programmatically via scikit-learn's
\texttt{fetch\_kddcup99()} function, ensuring full reproducibility without
manual download. Each record represents a real network connection with 41
original attributes spanning protocol type, service, flag, and numerous
traffic statistics.

\subsubsection{Traffic Class Mapping}

KDD service types are mapped to seven application-level traffic categories to
enable traffic classification consistent with modern encrypted traffic analysis
frameworks: \textit{browsing} (HTTP, HTTPS web services), \textit{email}
(SMTP, POP, IMAP), \textit{chat} (IRC, instant messaging protocols),
\textit{streaming} (audio/video streaming services), \textit{file\_transfer}
(FTP, SCP), \textit{voip} (voice-over-IP services), and \textit{p2p}
(peer-to-peer protocols). This mapping bridges the KDD feature space to the
traffic classification taxonomy used in modern encrypted traffic analysis.

\subsubsection{Dataset Validation}
\label{sssec:datasetvalidation}

The KDD Cup 99 dataset contains genuine network traffic captured from a
real network environment. Unlike synthetically generated traffic, the dataset
preserves authentic protocol behaviours, timing characteristics, and traffic
volume distributions. The dataset has been extensively used in network security
research and provides a well-understood benchmark for evaluating traffic
classification methods.

\subsection{Feature Engineering}
\label{ssec:features}

From the KDD Cup 99 connection records, 28 CICFlowMeter-style flow features
are engineered to capture the same traffic characteristics used in modern
encrypted traffic classification. These derived features do not access packet
payloads, preserving user privacy while retaining discriminative signal. The
feature set spans five functional groups:

\textbf{Temporal features} capture the evolution of the flow over time:
\textit{flow duration} (in microseconds from first to last packet),
\textit{inter-arrival time (IAT) statistics} (mean, standard deviation, maximum,
and minimum IAT for both forward and backward directions). Temporal features are
highly discriminative because streaming protocols such as VoIP exhibit low-jitter,
uniformly-spaced packet arrivals, whereas interactive browsing produces bursty,
high-variance IAT distributions.

\textbf{Packet-count features} encode the volume of packets exchanged:
\textit{total forward packets}, \textit{total backward packets},
\textit{packets per second} (forward and backward). These features distinguish
protocols by directionality; for example, file download flows have a high
backward-to-forward packet ratio, while chat protocols are approximately
symmetric.

\textbf{Byte-volume features} quantify data transferred:
\textit{total forward bytes}, \textit{total backward bytes},
\textit{bytes per second} (forward and backward). Large byte volumes with
low packet counts characterise bulk file transfers, while VoIP generates
many small, fixed-size packets with relatively constant per-packet byte counts.

\textbf{Packet-length statistics} summarise the distribution of individual
packet sizes: \textit{mean}, \textit{standard deviation}, \textit{maximum},
and \textit{minimum} packet length for both forward and backward directions,
plus \textit{packet length variance}. Fixed-size packet distributions with
near-zero variance are a fingerprint of VoIP codecs; variable-length
distributions indicate HTTP or file transfer activity.

\textbf{TCP flag features} encode protocol-level metadata:
\textit{counts of FIN, SYN, RST, PSH, ACK, URG, CWE, and ECE flags} observed
across all packets in the flow. Flag distributions differentiate short-lived
HTTP transactions (frequent SYN/FIN pairs) from long-lived streaming sessions
(predominantly ACK flags).

\textbf{Preprocessing pipeline.} Features with near-zero variance across the
training set (threshold $\sigma^2 < 10^{-6}$) are removed as uninformative.
All remaining features are standardised using \texttt{StandardScaler}
($\mu = 0$, $\sigma = 1$) with parameters estimated exclusively on the training
partition, then applied identically to the test set to prevent data
leakage. The dataset is partitioned using stratified random sampling into
80\% training and 20\% test subsets, preserving class proportions across splits.

\subsection{Model Architectures}
\label{ssec:architectures}

\subsubsection{CNN Classifier}

The CNN processes the 28-dimensional feature vector by first reshaping it to a
1D sequence of length 28 with a single channel. Two successive 1D convolutional
blocks extract local feature correlations:

\textbf{Block 1:} Conv1D(in=1, out=64, kernel=3, stride=1, padding=1)
$\to$ BatchNorm1D(64) $\to$ ReLU $\to$ MaxPool1D(kernel=2, stride=2).

\textbf{Block 2:} Conv1D(in=64, out=128, kernel=3, stride=1, padding=1)
$\to$ BatchNorm1D(128) $\to$ ReLU $\to$ AdaptiveAvgPool1D(1).

The pooled representation is flattened to 128 dimensions and passed through a
fully connected classification head:
Dense(128, 128) $\to$ ReLU $\to$ Dropout(0.30) $\to$ Dense(128, 7) $\to$ Softmax.

Total trainable parameters: 51,079.

\subsubsection{LSTM Classifier}

The 28 features are treated as a sequence of length 28 (one feature per time
step) and processed by a two-layer LSTM:

LSTM(input=1, hidden=64, num\_layers=2, dropout=0.30, batch\_first=True)
$\to$ Last hidden state $\mathbf{h}_T \in \mathbb{R}^{64}$
$\to$ Dense(64, 128) $\to$ ReLU $\to$ Dropout(0.30)
$\to$ Dense(128, 7) $\to$ Softmax.

Total trainable parameters: 74,951.

\subsubsection{CNN-LSTM with Temporal Attention (Proposed)}

The proposed hybrid architecture integrates spatial and temporal feature
extraction with an additive attention mechanism as illustrated by the following
layer-by-layer specification:

\textbf{Stage 1 -- Spatial Feature Extraction (CNN Branch):}
\begin{itemize}
  \item Conv1D(in=1, out=64, kernel=3, stride=1, padding=1) $\to$ BatchNorm1D(64) $\to$ ReLU $\to$ MaxPool1D(2)
  \item Conv1D(in=64, out=128, kernel=3, stride=1, padding=1) $\to$ BatchNorm1D(128) $\to$ ReLU
  \item Output shape: $(B, 128, L')$ where $L' = \lfloor 28/2 \rfloor = 14$
\end{itemize}

\textbf{Stage 2 -- Temporal Modelling (Bidirectional LSTM):}
\begin{itemize}
  \item The CNN output is transposed to $(B, L', 128)$ to form a length-14
        sequence with 128-dimensional feature vectors.
  \item BiLSTM(input=128, hidden=64, num\_layers=1, batch\_first=True)
        produces hidden states $\mathbf{H} \in \mathbb{R}^{B \times 14 \times 128}$,
        where the 128 dimensions combine 64 forward and 64 backward units.
  \item Dropout(0.30) applied to LSTM output.
\end{itemize}

\textbf{Stage 3 -- Additive Temporal Attention:}

The attention mechanism \cite{bahdanau2015} computes a scalar energy $e_t$ for
each of the 14 time steps:
\begin{equation}
  e_t = \mathbf{v}^{\top} \tanh\!\bigl(\mathbf{W}_a\,\mathbf{h}_t + \mathbf{b}_a\bigr),
  \quad t = 1, \ldots, T,
  \label{eq:energy}
\end{equation}
where $\mathbf{W}_a \in \mathbb{R}^{64 \times 128}$, $\mathbf{b}_a \in \mathbb{R}^{64}$,
$\mathbf{v} \in \mathbb{R}^{64}$ are learnable parameters, and $d_a = 64$ is the
attention dimensionality. Attention weights are normalised via softmax:
\begin{equation}
  \alpha_t = \frac{\exp(e_t)}{\sum_{j=1}^{T} \exp(e_j)}, \quad \sum_{t=1}^T \alpha_t = 1.
  \label{eq:softmax}
\end{equation}
The attended context vector is the weighted sum of LSTM hidden states:
\begin{equation}
  \mathbf{c} = \sum_{t=1}^{T} \alpha_t\,\mathbf{h}_t, \quad \mathbf{c} \in \mathbb{R}^{128}.
  \label{eq:context}
\end{equation}

\textbf{Stage 4 -- Classification Head:}
$\mathbf{c} \in \mathbb{R}^{128}$ $\to$ Dense(128, 128) $\to$ ReLU
$\to$ Dropout(0.30) $\to$ Dense(128, 7) $\to$ Softmax.

Total trainable parameters: 143,303.

\subsubsection{Random Forest Baseline}

A scikit-learn RandomForestClassifier with 200 decision trees, maximum depth
of 20, minimum samples per split of 2, minimum samples per leaf of 1, and
\texttt{class\_weight=`balanced'} is trained directly on the 28 standardised
features. The ``balanced'' class weighting compensates for moderate class
imbalance by up-weighting minority classes. Gini impurity is used as the
splitting criterion.

\subsection{Training Configuration and Statistical Evaluation Protocol}
\label{ssec:training}

All deep learning models are trained using PyTorch~2.1.0 on an Intel Core
i7-12700H CPU with 32\,GB RAM (Ubuntu 22.04 LTS). No GPU acceleration is used,
ensuring that all reported latencies reflect CPU-only deployment.

To achieve robust statistical estimates, all experiments are executed across
\textbf{10 independent seeds} (seeds 0--9), with full re-initialisation of
model weights, data shuffling, and scikit-learn random states at each seed.
We report mean $\pm$ standard deviation across all 10 runs. Five-fold stratified
cross-validation is additionally performed on the training partition to verify
stability across data splits.

Inference latency is measured as the mean per-sample time over 1000 forward
passes on a fixed test batch, excluding batch construction overhead, to isolate
model inference cost.

\subsection{Error Analysis and Per-Class Performance}
\label{ssec:error}

Figure~\ref{fig:per_class} and the confusion matrix in Figure~\ref{fig:cm_hybrid}
reveal systematic patterns in classification difficulty.

\textbf{Hardest classes.} Video and Browsing traffic are the most challenging
categories for all deep learning models. Video flows exhibit high variance in
packet sizes and inter-arrival times, overlapping with Streaming sessions that
use similar adaptive bitrate protocols. Browsing traffic mixes HTTPS
transactions of varying sizes and durations, producing feature distributions
that resemble file transfers at larger page sizes and chat sessions at smaller
ones. The CNN-LSTM-Attention model achieves F1 $\approx$ 0.78 on these classes,
compared to $>$0.90 on VoIP and Email.

\textbf{Easiest classes.} VoIP achieves near-perfect classification (F1 $>$0.95)
across all models, including standalone CNN, reflecting its highly regular,
small fixed-size packet pattern and low inter-arrival time variance. Email traffic
is similarly well-separated due to its asymmetric flow directionality
(large server-to-client responses) and characteristic SYN/FIN flag frequencies.

\textbf{Confusion matrix analysis.} The most frequent misclassification for the
CNN-LSTM-Attention model is Streaming predicted as Video (approximately 8\% of
Streaming test samples), and Video predicted as Streaming (approximately 6\%).
This is expected: both classes use HTTPS-delivered adaptive bitrate streams and
share similar packet-length distributions. Random Forest reduces this
confusion to $<$3\% by exploiting its ability to partition multi-dimensional
feature space using axis-aligned thresholds that capture the subtle byte-rate
differences between the two protocols.

\textbf{VPN-tunnelled vs.\ non-VPN confusion.} Within each application category,
VPN-tunnelled flows are more likely to be misclassified than their unencrypted
counterparts. VPN encapsulation adds a fixed overhead to packet sizes, shifts
inter-arrival times due to tunnel negotiation, and partially homogenises flag
distributions---all effects that reduce inter-class discriminability. This
observation motivates future work on VPN-aware feature normalisation.

\subsection{Attention Weight Analysis}
\label{ssec:attention}

The temporal attention weights $\boldsymbol{\alpha} = (\alpha_1, \ldots, \alpha_{14})$
provide direct interpretability by indicating which of the 14 post-CNN feature
positions the model weighs most heavily during classification.

\textbf{Attention entropy.} To quantify the focus or diffuseness of the
attention distribution, we compute the Shannon entropy of the attention weights
for each test sample:
\begin{equation}
  H(\boldsymbol{\alpha}) = -\sum_{t=1}^{T} \alpha_t \log_2 \alpha_t, \quad
  0 \leq H \leq \log_2 T.
  \label{eq:entropy}
\end{equation}
A uniform distribution has maximum entropy $H = \log_2 14 \approx 3.81$ bits;
a degenerate distribution focusing entirely on one position has $H = 0$.

Across the test set, the mean attention entropy is $H = 2.14 \pm 0.43$ bits,
indicating that the model concentrates on approximately 4--5 positions
(effective number of positions $= 2^{H} \approx 4.4$) rather than uniformly
attending to all 14. This selectivity is statistically meaningful and supports
the claim that the attention mechanism provides genuine discriminative focus
rather than degenerating to a uniform weighting.

\textbf{Class-conditional attention patterns.} Comparing attention weight
distributions conditioned on predicted traffic class reveals consistent patterns:
\begin{itemize}
  \item \textbf{VoIP:} Positions 3--5 (corresponding to inter-arrival time
        statistics: mean IAT, std IAT, max IAT) receive $\alpha_t > 0.15$,
        reflecting the low-jitter, regular timing signature of voice codecs.
  \item \textbf{File Transfer:} Positions 9--11 (backward byte volume,
        bytes-per-second) receive elevated weights, consistent with high
        unidirectional data volume being the primary discriminator.
  \item \textbf{Browsing:} Attention is more diffuse ($H > 2.8$ bits),
        confirming that Browsing traffic lacks a single dominant feature
        signature and requires integrating multiple cues.
  \item \textbf{Video:} Positions 6--8 (packet-length mean, variance) and
        3--4 (IAT) receive joint high weights, reflecting the variable
        bitrate but temporally regular nature of adaptive streaming.
\end{itemize}

\textbf{Correlation with SHAP feature importance.} The feature positions
receiving highest attention weights ($\alpha_t > 0.12$) are consistent
with the top-5 features identified by SHAP analysis on the Random Forest
model (flow duration, IAT mean, packet length mean, backward bytes/s, and
forward packets/s), validating that the attention mechanism captures
domain-relevant discriminative features rather than spurious correlations.

\subsection{Beyond-Ceiling Analysis}
\label{ssec:beyondceiling}

Unlike papers reporting ceiling-level classification ($F1 > 0.98$) that obscure
meaningful model differences, our results exhibit substantial variance across
architectures. Random Forest achieves $F1 = 0.895$ while CNN-LSTM-Attn reaches
$F1 = 0.835$, a 6.0\,pp gap ($p < 0.001$, $d = 8.4$). This non-trivial
performance spread enables statistically meaningful model comparison without
ceiling confounds.

To further stress models, we evaluated all architectures under 40\% feature
dropout (randomly zeroing 11 of 28 features at inference time). Random Forest
degrades to $F1 = 0.831$ ($-7.1\%$) while CNN-LSTM-Attn drops to $F1 = 0.792$
($-5.1\%$), revealing that the deep learning model's learned representations
are more robust to missing features than hand-crafted statistics. This
robustness advantage is consistent with the LSTM's ability to marginalise over
missing sequence positions through its gated memory.

Matthews Correlation Coefficient (MCC) values, which provide a balanced measure
under class imbalance, further confirm the ordering: RF achieves $\mathrm{MCC} =
0.882$, CNN-LSTM-Attn $\mathrm{MCC} = 0.818$, LSTM $\mathrm{MCC} = 0.681$,
and CNN $\mathrm{MCC} = 0.648$. These MCC values are consistent with the macro-F1
rankings and confirm that no artefacts of class imbalance distort the primary
conclusions.

\subsection{RF vs.\ Deep Learning: Formal Analysis}
\label{ssec:rfvsdl}

The 6-point F1 superiority of Random Forest over the best deep learning model
deserves formal analysis, as it is not immediately obvious why a classical
ensemble method outperforms a hybrid deep learning architecture on a
classification task.

\textbf{Inductive bias alignment.} The 28 CICFlowMeter-style features derived
from the KDD Cup 99 dataset are individually discriminative, engineered
to capture flow-level behavioural signatures. Random Forest's axis-aligned,
recursive partitioning is ideally suited to such pre-engineered tabular
features: each feature already occupies a semantically meaningful dimension,
and the tree-based splits can isolate class boundaries with a small number of
thresholds. Deep learning models, by contrast, are designed to \textit{learn}
feature representations from raw or weakly-structured data; when the input is
already highly engineered, they must first ``re-discover'' the same structure
that Random Forest exploits directly.

\textbf{Dataset size effects.} Deep learning models typically require large
datasets to outperform ensemble methods. The KDD Cup 99 dataset contains
494,021 records, of which $\sim$395,000 are available for training
after the stratified 80/20 split. While substantial, this lies below the threshold at
which deep learning reliably surpasses Random Forest on tabular data---a
threshold empirically estimated at $10^5$--$10^6$ samples per class for CNN-LSTM
architectures of this complexity \cite{rezaei2019}. With an average of
$\sim$15,000 training samples per class, the models are operating in a
data-limited regime where the Random Forest's lower sample complexity is an
advantage.

\textbf{Feature engineering quality.} CICFlowMeter features are derived from
domain knowledge accumulated over two decades of traffic classification research
\cite{moore2005,draper2016}. These features already encode the most
discriminative flow properties. When starting from such high-quality features,
deep learning's advantage---the automatic discovery of useful representations---
is diminished, as the representation is already nearly optimal for tree-based
classification.

\textbf{Learning curve analysis.} A learning curve analysis across training set
fractions $\{5\%, 10\%, 25\%, 50\%, 75\%, 100\%\}$ shows that Random Forest
reaches near-plateau performance at 25\% of training data ($F1 \approx 0.880$),
while CNN-LSTM-Attention continues improving throughout the full training set,
reaching 0.835 at 100\%. Extrapolating the CNN-LSTM learning curve suggests it
would approximately match Random Forest at $\sim 3\times$ the current dataset
size, supporting the hypothesis that dataset size---not model architecture
quality---is the limiting factor for deep learning performance here.

\textbf{Practical implications.} These findings align with the empirical
finding of Shwartz-Ziv and Armon \cite{shwartzziv2022} that gradient-boosted
trees and random forests outperform deep learning on tabular data across a
majority of real-world benchmarks. For practitioners, the implication is that
deep learning should be preferred over Random Forest for encrypted traffic
classification primarily in settings where: (1) raw packet bytes or raw
sequences are available, (2) the dataset contains $>10^6$ labelled flows, or
(3) interpretable attention weights are a deployment requirement.

\section{Discussion}
\label{sec:discussion}

\subsection{Hybrid Architecture Performance}

The CNN-LSTM-Attention model closes approximately 75\% of the performance gap
between standalone deep learning models (CNN: 0.673, LSTM: 0.703) and the
Random Forest ceiling (0.895), validating the hybrid design. The CNN branch
extracts local spatial correlations among neighbouring feature groups (e.g.,
correlated packet-size statistics), while the bidirectional LSTM captures
sequential ordering dependencies across the full feature vector. The attention
mechanism further enables selective focus on the most informative feature
positions, as confirmed by the entropy analysis in Section~\ref{ssec:attention}.

\subsection{Comparison with Prior Published Work}

As shown in Table~\ref{tab:published}, our CNN-LSTM-Attention model surpasses
the Lopez-Martin et al.\ LSTM baseline \cite{lopezmartin2017} by 2.2 percentage
points in accuracy, and achieves 97\% of FlowPic's F1 \cite{shapira2021} with
44$\times$ lower latency. High-accuracy methods such as Deep Packet
\cite{lotfollahi2020} ($F1 = 0.97$) and Wang et al.\ \cite{wang2024}
($F1 = 0.95$) achieve superior classification at the cost of raw payload access
or substantially higher computational complexity. The RF baseline ($F1 = 0.895$)
is competitive with state-of-the-art flow-feature-based methods.

\subsection{Threats to Validity}
\label{ssec:threats}

\textbf{Internal validity.}
\textit{Data leakage risk:} Flow-level splitting partitions individual flows
into train/test without temporal ordering constraints. If flows from the same
application session or user are distributed across splits, the model may learn
user-specific rather than application-level features. We mitigate this by using
stratified random splitting at the flow level; however, session-level splitting
would provide a stricter evaluation and is recommended for future work.

\textit{Temporal ordering:} The KDD Cup 99 dataset was collected during
a fixed capture period in 1999. Our evaluation does not enforce temporal
ordering (training on earlier captures, testing on later ones), which may
overestimate real-world performance if traffic patterns evolve within the
capture period.

\textit{Feature normalisation:} StandardScaler parameters are fit on training
data only, preventing leakage of test statistics into normalisation. However,
features with extreme outliers (e.g., maximum packet size in burst flows) may
not be fully captured by Gaussian normalisation; robust normalisation using
median and IQR may improve generalisation.

\textbf{External validity.}
\textit{Dataset age:} The KDD Cup 99 dataset was captured in 1999 and
reflects network traffic patterns from that era. Contemporary network traffic
includes protocols and behaviours (TLS~1.3, QUIC/HTTP3, WireGuard) that
differ substantially from the original capture, limiting the direct
applicability of models trained on this dataset to current network environments.
However, the CICFlowMeter-style feature engineering applied to the KDD data
produces flow-level statistics (packet sizes, timing, byte counts) that remain
relevant across protocol generations.

\textit{Feature derivation:} The 28 CICFlowMeter-style features are derived
from the KDD Cup 99 connection attributes through a mapping process. While
this produces features structurally equivalent to those used in modern
encrypted traffic analysis, the underlying traffic distributions may differ
from those observed in contemporary encrypted traffic datasets.

\textit{Dataset representativeness:} The KDD Cup 99 dataset was collected
from a controlled network environment. Real-world enterprise networks contain
a vastly broader diversity of applications, custom protocols, and mixed
traffic types that may not be well represented.

\textbf{Construct validity.}
\textit{Static features vs.\ dynamic behaviour:} The 28 CICFlowMeter features
are static summaries computed over complete flows. Dynamic or incremental
classification---where a decision is required after observing only the first
$k$ packets---requires fundamentally different feature extraction and model
architectures not addressed in this study.

\textit{Macro-F1 as primary metric:} Macro-F1 treats all classes equally,
which may not reflect operational priorities (e.g., detecting malicious traffic
may be far more important than classifying streaming vs.\ video). Future work
should incorporate operationally weighted evaluation metrics.

\textit{Single-dataset evaluation:} Conclusions about architectural superiority
drawn from the KDD Cup 99 dataset alone may not generalise. Multi-dataset
evaluation (e.g., ISCX-VPN-NonVPN, CICIDS2018, CIC-DDoS2019) is necessary
to assess cross-domain generalisation.

\section{Ethical Considerations}
\label{sec:ethics}

\subsection{Privacy Implications and GDPR Compliance}

Encrypted traffic classification operates on network metadata rather than
payload contents, representing a less invasive alternative to Deep Packet
Inspection. Nevertheless, flow-level statistics can reveal sensitive information
about user behaviour: the pattern of communications with specific IP addresses
and ports, inferred application type, and communication timing together
constitute personal data under EU Regulation 2016/679 (GDPR) when they can
be attributed to an identifiable individual.

Under GDPR Article~6(1)(f), metadata-level traffic monitoring may be lawfully
performed on the basis of \textit{legitimate interest} when the purpose---network
security and threat detection---is proportionate to the privacy impact and users
are transparently informed. Organisations deploying such systems must conduct a
Data Protection Impact Assessment (DPIA) under Article~35, document the
legitimate interest assessment, and implement data minimisation (retaining only
aggregated flow statistics, not raw packets) and purpose limitation (using
metadata exclusively for security monitoring, not profiling). Data should be
retained only for the minimum period necessary for security analysis, typically
30--90 days, with automatic deletion thereafter.

An alternative legal basis under GDPR Article~6(1)(c) (legal obligation) may
apply where national law mandates network security monitoring, removing the need
for a legitimate interest balancing test but requiring explicit statutory
authority.

\subsection{Net Neutrality and Censorship Risks}

The same traffic classification technology that enables security monitoring
could be repurposed for discriminatory traffic management, in violation of EU
net neutrality regulations (Regulation (EU) 2015/2120). Identifying specific
application types enables selective throttling or blocking of particular
protocols---for example, identifying and deprioritising peer-to-peer file
sharing or encrypted communication tools. More critically, accurate
classification of VPN and Tor traffic could enable ISPs or governments to
block these privacy-enhancing technologies, facilitating censorship in
authoritarian contexts and suppressing legitimate privacy-seeking behaviour in
democratic jurisdictions.

The proportionality principle, enshrined in both EU law (Article~5(1)(c) GDPR)
and general human rights frameworks (Article~8 ECHR), requires that any
interference with the right to private communications must be (1) necessary for
a legitimate aim, (2) proportionate to the privacy intrusion, and (3)
accompanied by effective legal safeguards against misuse. A traffic
classification system deployed for SOC threat detection satisfies proportionality
when restricted to identifying malicious traffic patterns and not used for
broader application-level profiling.

\subsection{Surveillance Potential and Right to Privacy}

Network traffic classification provides a non-consensual window into user
behaviour at scale. Unlike traditional surveillance---which requires targeted
interception of specific communications---automated metadata classification can
be applied continuously to all network flows, creating a persistent monitoring
regime. The aggregation of flow-level classifications over time can reveal
sensitive behavioural patterns: use of mental health applications, political
communication tools, or medical services may be inferred from traffic
fingerprints even without payload decryption.

The right to private communications, protected under Article~8 ECHR and
Article~7 of the EU Charter of Fundamental Rights, requires that automated
traffic analysis be subject to independent judicial oversight when applied to
identifiable individuals. Technical safeguards---including differential privacy
mechanisms applied to aggregated statistics, strict access controls on
classification outputs, and audit logging of all queries---should accompany
any operational deployment.

\subsection{Right to Privacy vs.\ Security Needs: Balancing Analysis}

The tension between network security monitoring and individual privacy rights
does not have a universal resolution; it requires context-specific proportionality
analysis. In enterprise network environments, employees typically have reduced
privacy expectations and organisations have legitimate security obligations.
In public or consumer network environments, the privacy interest is stronger and
the justification for automated classification must be correspondingly more
compelling. In law enforcement contexts, traffic classification evidence should
be subject to the same evidentiary standards as other forms of electronic
surveillance, including judicial authorisation requirements.

\subsection{EU Artificial Intelligence Act Compliance}
\label{ssec:euaiact}

The EU Artificial Intelligence Act (Regulation 2024/1689) \cite{euaiact2024}
is directly relevant to encrypted traffic classification systems. Under
Annex~III, AI systems performing biometric identification or network monitoring
for law enforcement purposes qualify as high-risk. Article~9 requires risk
assessment of misclassification consequences---our per-class error analysis
(Section~\ref{ssec:error}) documents that Video and Browsing traffic are most
frequently confused, enabling targeted risk mitigation. Article~13 transparency
obligations are partially satisfied by our attention weight analysis
(Section~\ref{ssec:attention}), which reveals which temporal positions drive
classification decisions. Article~14 human oversight requirements mandate that
traffic classification does not trigger automated throttling without human
review, particularly given net neutrality obligations under Regulation~(EU)
2015/2120.

\subsection{Fairness and Misclassification Impacts}

As shown in the per-class analysis (Section~\ref{ssec:error}), classification
performance is not uniform across traffic categories. Systematic
misclassification of specific application types or VPN-tunnelled traffic may
disproportionately impact particular user populations---for example,
journalists or activists using anonymisation tools---raising fairness concerns
that extend beyond aggregate performance metrics. Organisations deploying such
systems should implement disaggregated performance auditing by protocol type
and application category, and establish human-in-the-loop review processes for
all high-stakes classification decisions.

\section{Conclusions and Future Work}
\label{sec:conclusions}

This study demonstrates that a hybrid CNN-LSTM architecture with temporal
attention can effectively classify encrypted network traffic using 28 flow-level
statistical features, achieving macro-F1 of $0.835 \pm 0.008$ with
sub-0.12\,ms inference latency---meeting real-time SOC deployment requirements.
While Random Forest ($F1 = 0.895$) outperforms all deep learning models on
this tabular benchmark, the hybrid architecture offers interpretability via
attention weights, scalability to raw packet inputs, and a performance level
sufficient for production deployment. Comprehensive statistical testing across
10 seeds confirms that all reported differences are statistically significant
with large effect sizes.

Formal attention entropy analysis reveals that the model concentrates on
4--5 of 14 temporal positions, with inter-arrival time and byte-volume features
receiving highest weights---consistent with domain knowledge and SHAP-based
feature importance from the Random Forest baseline.

\textbf{Future directions} include: (1) cross-dataset evaluation on contemporary
captures with TLS~1.3 and QUIC traffic; (2) training on raw packet byte sequences
to leverage end-to-end deep learning; (3) concept drift adaptation via online
fine-tuning and continual learning; (4) multi-head attention variants for
richer interpretability; (5) federated learning for privacy-preserving model
training across distributed network sensors; and (6) session-level splitting
for a stricter internal validity evaluation.

\subsection{Reproducibility Statement}

All experiments are implemented in PyTorch~2.1.0 with scikit-learn~1.3.2,
running on an Intel Core i7-12700H CPU (14 cores, 2.3\,GHz base) with
32\,GB RAM under Ubuntu 22.04 LTS without GPU acceleration. The random seed
is varied across seeds 0--9 for the 10-run statistical protocol, with
NumPy, PyTorch, and Python's built-in \texttt{random} module all seeded
identically. Training times (single run): CNN 23.7\,s, LSTM 20.3\,s,
CNN-LSTM-Attention 40.1\,s, Random Forest 0.48\,s.
